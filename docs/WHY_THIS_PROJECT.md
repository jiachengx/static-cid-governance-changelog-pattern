# Why this project

A normal changelog says what changed. A governance changelog also explains why a change was made, what risk it reduces, which boundary it affects, and what future maintainers need to know before modifying related rules.

For nonprofit and static publishing projects, this matters because maintainers may change over time. Without governance notes, future teams may see only code changes but not the reasoning behind identity rules, public/private boundaries, migration steps, or release accountability.

This repository provides a reusable changelog pattern for:

- governance-oriented release notes;
- nonprofit maintenance handoff;
- public/private boundary tracking;
- CID/hash snapshot change notes;
- reference consistency change notes;
- release-gate validation records;
- long-term project memory.

The goal is to help organizations keep their public technology governance understandable, auditable, and maintainable.
